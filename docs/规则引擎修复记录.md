# 规则引擎修复记录

> 记录一次对**统一规则引擎（`TradeX.Rules`）+ 策略决策内核 + 回测引擎**的正确性 / 完整性审查，
> 以及据此做出的修复与设计决策。按"问题 → 根因 → 修复"组织，供回归排查与后续演进参考。
>
> 范围约定：本次按**最新设计**推进，**不再兼容旧的 `EntryCondition` / `ExitCondition` 双条件策略**，
> 策略统一以 `ExecutionRule`（规则集 JSON）为唯一决策来源。
>
> 时区约定沿用全局 UTC；回测以 K 线时间戳为模拟时钟。

---

## 1. 背景

规则引擎将策略定义（`ExecutionRule` 规则集 JSON）解析为 `RuleSet`，
每个 tick 合并指标后按优先级遍历规则，逐条经"上下文 → 约束 → 条件"三道闸，
首条命中即产出动作并短路返回，再映射为统一决策交由实盘 / 回测各自兑现。
该引擎取代了旧的 `ConditionTreeEvaluator` 与 `VolatilityGridExecutor`。

审查目标：确认迁移后的求值语义忠实，且不存在会导致**错误交易**或**回测失真**的缺陷。

## 2. 发现的问题与修复

### 高危

| # | 问题 | 根因 | 影响 | 修复 |
|---|------|------|------|------|
| 1 | 触发追踪作用域过宽 + 用墙钟时间 | `TriggerTracker` 以裸 `ruleCode` 为键、内部读 `DateTime.UtcNow` | 不同策略/交易对的同名规则共享 `MinInterval` 冷却；回测毫秒级跑完导致带 `minInterval` 的规则整场只触发一次 → 回测系统性失真 | `ITriggerTracker` 改为 `(key, now)` 签名，不读墙钟；键由 evaluator 按 `{ScopeKey}/{ruleCode}` 组合。实盘 `ScopeKey={bindingId}:{pair}` + `IClock.UtcNow`；回测每次 `Run` 用唯一 `backtest:{guid}` + K 线时间 |
| 2 | 单条规则解析失败被静默丢弃（fail-open） | `RuleSetParser` 捕获异常后跳过该规则、其余照常生效 | 出场规则坏掉而入场正常 → 策略"只买不卖" | 改为 **fail-closed**：任一规则解析失败即拒绝整个规则集（返回 `null` → `Hold`） |
| 3 | `ExecutionRule` 在 API 层从未校验 | `RuleSetValidator` 已注册 DI 但无控制器调用，控制器只校验旧 Entry/Exit | 坏 JSON 静默入库，运行时退化为 `Hold` 或半套生效 | 控制器 Create/Update 改走 `RuleSetValidator`；`/validate` 端点改校验规则集 |
| 4 | 校验器与求值器对上下文指标不一致 | `DEVIATION_FROM_AVG` 等上下文指标未注册进 `IIndicatorRegistry`，校验器报"未注册" | 接通 #3 后所有合法网格/加仓规则反而保存不了 | `ContextIndicators.All` 暴露集合，`ConditionTreeValidator` 识别"技术指标 ∪ 上下文指标" |

### 中危

| # | 问题 | 根因 | 影响 | 修复 |
|---|------|------|------|------|
| 5 | 嵌套 `TRUE` 求值为 false | 仅在条件树**根节点**处理 `TRUE`，嵌套节点落入叶子分支 | 校验器允许的嵌套 `TRUE` 在运行时恒假 | `EvaluateNode` 在任意层级优先处理 `TRUE` |

## 3. 关键设计决策

### 3.1 仓位 `sizeType`：保留 `fixed` / `multiplier`，移除 `percent` / `grid`

- `fixed`（绝对 quote 金额）与 `multiplier`（`Size × 引用指标`，用于金字塔加仓）都是**纯函数**，
  在回测与实盘行为完全一致，size 在 evaluator 内即被解析为绝对 quote 金额。
- `percent`（按账户权益百分比）需要账户权益上下文：回测有 `cash` 可用，
  但**实盘需逐 tick 拉取余额**（鉴权调用，rate-limit 隐患），且会造成**回测/实盘背离**——
  破坏纯函数、无外部资金依赖的引擎模型，故移除。
- `grid` 无明确语义，移除。
- 二者均被 `RuleSetValidator` 拒绝（保存即报错），消除"声称某单位但从不兑现"的隐患。
- 若将来确需 `percent`，应通过 **binding 上的资金分配字段**显式提供资本，而非引擎隐式拉取，
  作为独立特性推进。

### 3.2 移除 `EntryCondition` / `ExitCondition`（彻底）

- 从 `Strategy` 模型、领域事件、UseCase/DTO/命令、控制器、`BacktestService` 守卫全链路移除。
- 删除已失效的 `LegacyStrategyScanner`（专扫旧条件树语法）及其 Admin 端点。
- 生成 EF 迁移 `20260610145350_DropStrategyEntryExitConditions` 删除 `entry_condition` / `exit_condition` 两列。
  > ⚠️ **部署注意**：库与 Go 实盘 Worker 共享。该迁移为破坏性 drop，**未自动应用**；
  > 确认 Go 端不再读这两列后，再在部署窗口手动应用。

### 3.3 回测多笔持仓（FIFO lot）重写

`BacktestEngine` 由单一持仓改为 FIFO `lot` 队列：

- `buy`（`EnterMarket`/`AddGridLot`）→ 追加一笔 lot，加权更新均价，支持持仓中加仓；
- `ReduceOneLot` → 平掉最早一笔 lot；
- `SellAll` → 逐笔平掉全部 lot；
- 末根强制平掉剩余持仓，保证账目完整。

单笔持仓策略结果与旧引擎一致（parity / 回归测试锁定）；新增金字塔加仓回测用例。

### 3.4 触发追踪的作用域与时钟

- 实盘：`ITriggerTracker` 进程级单例，键 `{bindingId}:{pair}/{ruleCode}`，时间用注入的 `IClock.UtcNow`。
- 回测：与实盘**进程隔离**（`AddTradingWorker` vs `AddTradingBacktestWorker`），
  且每次 `Run` 用唯一 `backtest:{guid}` 作用域键，避免同进程多回测任务间 `MinInterval` 冷却串扰；
  时间用当前 K 线时间戳。

## 3.5 第二轮完整性修复

复审后补齐的完整性缺口：

- **`sell` 支持按金额减仓**：`Sell` 动作的 `Size`（含 `multiplier` 解析结果）现透传为 `StrategyAction.Reduce` 的目标 quote 金额——自最早持仓起按 **整笔 lot 粒度 FIFO** 平仓，累计名义价值达到目标即止；`size` 为 0 时退化为"平最早一笔"。实盘（`SelectPositionsToReduce`）与回测同语义。
- **规则 `code` 唯一性**：`RuleSetValidator` 强制规则集内 `code` 唯一（`code` 是触发追踪键与日志标识，重复会致 MinInterval 冷却串扰）。无正当重复场景，零运行时代价。
- **`multiplier` 必须带 `sizeMultiplierRef`**：`sizeType=multiplier` 时校验器要求提供 `sizeMultiplierRef` 且引用指标已知（技术或上下文），杜绝静默退化为 fixed。`ConditionTreeValidator.IsKnownIndicator` 公开供复用。
- **入场兜底金额具名化**：实盘 `EnterMarket` 无 size 兜底由魔数 `100m` 提取为常量 `FallbackEntryQuoteSize`，命中即记 Warning（提示规则缺 size）。
- **移除死枚举 `AddGridLot`**：`Buy` 恒映射 `EnterMarket`；`ReduceOneLot` 重命名为 `Reduce`（语义=按 size 或单笔减仓）。

## 3.6 第三轮复审修复

- **按金额减仓在 `price <= 0` 时的误平全部**（边界正确性 bug）：reduce-by-size 以"累计名义价值 = 数量 × 价格"判停；价格不可用时累计恒为 0、永不达标 → 会平掉**全部**持仓。实盘 `SelectPositionsToReduce` 与回测 Reduce 分支均加守卫：金额无效或 `price<=0` 时退化为"只平最早一笔"。

- **卖单在途幂等闸**（与买单对称，原缺失）：买单有进程内 `_inFlightEntry` 锁 + DB `HasActiveBuyAsync` 兜底，卖单原本没有任何保护——高频 Trade 路径上，卖单成交、持仓状态翻转前同一持仓会被相邻 tick 重复选中下卖单（双重平仓/过度卖出）。补 `_inFlightExit`（按持仓 Id）+ `IOrderRepository.HasActiveSellAsync`（该持仓的 Sell 且 Pending/PartiallyFilled）。`CloseGridPositionAsync` 拆分出 `CloseGridPositionCoreAsync`，幂等闸包裹于外层。新增 `OrderRepositoryTests` 覆盖该查询。

- **size 类字段仅限 buy/sell**（配置卫生，对齐"不接受会被静默忽略的配置"原则）：`sellAll`（全平）/ `hold`（不动）运行时忽略 `size`/`sizeType`/`sizeMultiplierRef`，校验器现对其上出现这些字段报错；`sell` 带 size（按金额减仓）合法。

## 4. 刻意保留的低危项（已在代码注释标注）

- `RecordTrigger` 仍在 evaluator 内完成：风控拒单后该规则仍进入 `MinInterval` 冷却。抽出会给实盘路径增加跨切面复杂度，暂留。
- `MaxPositionValue` 以**当前**名义价值判定（非加仓后投影值）。
- `multiplier` 的 `Math.Max(1, ·)` 钳制：无法表达 <1 缩仓倍率，且为线性放大（非指数 martingale）。
- `POSITION_PNL_PCT` 与 `DEVIATION_FROM_AVG` 计算等价（冗余但保留，避免破坏引用旧名的策略）。
- `RuleSet.Params` 已解析但暂未被消费。
