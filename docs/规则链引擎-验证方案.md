# 规则链引擎 — 验证方案

> 本文档定义规则链引擎的测试覆盖策略、预提交验证钩子及 CI 集成方案。

### 1.1 当前覆盖情况

### 1.2 需要补充的测试

以下测试区域需要**继续扩展**以完善覆盖率：

#### 1.2.1 ChainEngine 执行测试

测试单条链的执行流程和错误策略：

| 测试场景 | 说明 | 优先级 |
|---------|------|--------|
| 正常执行 | Gate → Filter → Derive → Size → Action → Risk → Override 完整管线 | 高 |
| Gate 阻断 | Gate 节点设 Blocked=true → 后续节点不执行 | 高 |
| Action 终止 | Action 节点异常 → Terminated=true → 后续节点不执行 | 高 |
| Filter/Skip 异常 | Filter 节点异常 → continue → 后续节点执行 | 高 |
| Risk 保护 | Risk 节点异常 → Actions=[] → 继续执行 | 高 |
| Derive 追加 | 多个 Derive 节点产出不同的 DerivedValues | 中 |
| Size 追加 | 多个 Size 节点都追加 SizeDecision | 中 |
| Override 覆盖 | Override 节点清空 Actions + Terminated | 高 |
| 混合决策链 | Action 产出 BUY+SELL+SELL_ALL 混合场景 | 中 |

#### 1.2.2 ChainCoordinator 合并测试

| 测试场景 | 说明 | 优先级 |
|---------|------|--------|
| 单链 HOLD | 链未产出任何 BUY/SELL → 输出 HOLD | 高 |
| 多链 BUY | 多条链都产 BUY → 按 orderType 聚合 | 高 |
| BUY + SELL 冲突 | 同 pair 有 BUY 和 SELL → 保留高 priority 侧 | 高 |
| SELL_ALL 覆盖 | 任一链产 SELL_ALL → 丢弃所有 BUY/SELL | 高 |
| 跨 pair 隔离 | 不同 pair 的决策独立合并 | 高 |
| SELL 超过持仓 | 累计卖出量 clamp 到持仓量 | 中 |
| 空列表 | 无任何 Actions → 返回空 | 中 |
| Blocked 链排除 | Blocked 链的 Actions 不参与合并 | 高 |
| Terminated 链排除 | Terminated 链的 Actions 不参与合并 | 高 |
| HOLD 回退 | 某链 HOLD 且无 BUY/SELL → 输出 HOLD | 高 |

#### 1.2.3 有状态节点测试

| 测试场景 | 说明 | 优先级 |
|---------|------|--------|
| dca_action 跨轮间隔 | 写入 lastAt → 第二轮回调不执行 | 高 |
| cooldown 冷却期 | 写入 lastTradeAt → 冷却期内阻止 | 高 |
| cost_anchored_rebalance 锚定 | 持久化 anchoredCost → 跨轮使用 | 中 |
| 无 StateStore 降级 | StateStore 为 null → 有状态节点降级行为 | 中 |

#### 1.2.4 ChainWorkerPool 测试

| 测试场景 | 说明 | 优先级 |
|---------|------|--------|
| Entry 队列满时丢弃 | BoundedChannel DropWrite → TrySubmitEntry 返回 false | 高 |
| Exit 队列阻塞 | BoundedChannel Wait → SubmitExitAsync 等待 | 中 |
| 优先消费 Exit | 同时有 Entry 和 Exit → 先执行 Exit | 中 |
| 单飞锁并发 | 同一 scopeKey 的第二次评估被拒绝 | 中 |
| Dispose 清理 | DisposeAsync 取消所有 Worker | 低 |

#### 1.2.5 边界与异常测试

| 测试场景 | 说明 | 优先级 |
|---------|------|--------|
| JSON 参数缺失 | 节点 params 为空/缺失字段 → 安全降级 | 高 |
| 信号缺失 | 节点依赖的信号不存在 → 跳过 | 高 |
| 价格为零或负 | `currentPrice <= 0` → 安全处理 | 中 |
| Size 为零 | `amount <= 0` → 不产出 SizeDecision | 中 |
| 持仓为空 | `Position` 为 null → Action 节点跳过卖出 | 中 |
| 组合为空 | `Portfolio` 为 null → Size 节点使用 `TotalEquity=0` | 中 |
| 超时取消 | `CancellationToken` 取消 → 线程安全终止 | 中 |

private static ChainState CreateState(
    Dictionary<string, decimal>? signals = null,
    decimal currentPrice = 50000m)
{
    var sigs = new Dictionary<string, Signal>();
    if (signals is not null)
        foreach (var (k, v) in signals)
            sigs[k] = new Signal { Name = k, Value = v };
    return new ChainState
    {
        Signals = sigs,
        Context = new EvalContext
        {
            Pair = "BTC/USDT", CurrentPrice = currentPrice,
            Portfolio = new PortfolioSnapshot { TotalEquity = 10000, AvailableCash = 5000 },
            Position = null, ScopeKey = "test|BTC/USDT",
        },
    };
}

// JSON 转 JsonElement
private static JsonElement ToJsonElement(string json)
    => JsonSerializer.Deserialize<JsonElement>(json);

## 3. 测试执行方式

### 3.1 本地运行

```bash
# 运行所有规则链测试
dotnet test backend/TradeX.Tests --filter "FullyQualifiedName~Rules"

# 运行特定测试类
dotnet test backend/TradeX.Tests --filter "FullyQualifiedName~ChainNodeTests"

# 运行特定测试方法
dotnet test backend/TradeX.Tests --filter "FullyQualifiedName~RegimeGate_ShouldBlock"

# 查看详细输出
dotnet test backend/TradeX.Tests --filter "FullyQualifiedName~Rules" -v n
```

### 3.2 构建验证

```bash
# 全量构建 + 测试
dotnet build backend/TradeX.sln && dotnet test backend/TradeX.Tests
```

## 4. 预提交验证 Hooks

### 4.1 验证脚本

在项目根目录创建 `.githooks/pre-commit`（可选，团队内部使用）：

```bash
#!/bin/bash
# 规则链引擎预提交验证

echo "=== 规则链测试 ==="
dotnet test backend/TradeX.Tests --filter "FullyQualifiedName~Rules" --no-restore
if [ $? -ne 0 ]; then
    echo "规则链测试失败，请修复后再提交"
    exit 1
fi

echo "=== 全量构建 ==="
dotnet build backend/TradeX.sln --no-restore
if [ $? -ne 0 ]; then
    echo "构建失败，请修复后再提交"
    exit 1
fi

echo "✓ 预提交验证通过"
```

### 4.2 安装方式

```bash
git config core.hooksPath .githooks
chmod +x .githooks/pre-commit
```

### 4.3 验证清单

| 检查项 | 命令 | 失败后果 |
|--------|------|---------|
| 规则链单元测试 | `dotnet test --filter "~Rules"` | 阻止提交 |
| 全量构建 | `dotnet build` | 阻止提交 |
| 新增节点注册 | 手动检查 `NodeRegistration.RegisterAllNodes()` 是否有追加 | 运行时节点找不到 |
| JSON schema 兼容 | 修改 `ChainDefinition` 或 `NodeInstance` 时检查前后端一致性 | 解析失败 |

## 5. CI 集成

### 5.1 GitHub Actions 配置

在 `.github/workflows/ci.yml` 中集成规则链专项测试：

```yaml
name: TradeX CI

on:
  push:
    branches: [main, develop]
  pull_request:
    branches: [main]

jobs:
  test:
    runs-on: ubuntu-latest

    steps:
      - uses: actions/checkout@v4

      - name: Setup .NET
        uses: actions/setup-dotnet@v4
        with:
          dotnet-version: '10.0.x'

      - name: Restore
        run: dotnet restore backend/TradeX.sln

      - name: Build
        run: dotnet build backend/TradeX.sln --no-restore

      - name: Run all tests
        run: dotnet test backend/TradeX.Tests --no-build --verbosity normal

      - name: Run rule chain specific tests
        run: dotnet test backend/TradeX.Tests --no-build --filter "FullyQualifiedName~Rules" --verbosity detailed
```

### 5.2 CI 验证矩阵

| 阶段 | 耗时预估 | 触发条件 |
|------|---------|---------|
| 规则链单元测试 | < 30s | 所有 PR |
| 全量测试套件 | < 2min | 所有 PR |
| 集成构建 | < 1min | merge 到 main |
| 代码覆盖率上报 | 附加 | 可选 |

### 5.3 覆盖率目标

## 6. 新增节点的验证流程

当新增一个规则节点时，必须完成以下验证步骤：

### 6.1 实现检查清单

```
[] 1. 节点类实现 IRuleNode 接口
[] 2. 实现 Kind (唯一标识)
[] 3. 实现 Phase (正确阶段枚举)
[] 4. 实现 Deps (信号依赖声明)
[] 5. 实现 ProcessAsync (核心逻辑)
[] 6. 参数类使用 record 声明
[] 7. 节点在对应的 *Nodes.cs 注册方法中注册
[] 8. NodeRegistration.RegisterAllNodes 覆盖到
```

### 6.2 测试检查清单

```
[] 1. 正向测试（正常执行、预期输出）
[] 2. 边界测试（空参数、缺失信号、零值）
[] 3. 异常测试（params 被篡改、依赖缺失）
[] 4. 有状态测试（如需持久化）
[] 5. 无 StateStore 降级测试（如适用）
```

### 6.3 文档检查清单

```
[] 1. 更新 docs/规则链引擎设计.md 的节点清单表
[] 2. 更新 docs/规则链引擎-配置指南.md 的参数说明表
[] 3. 在配置指南中增加 JSON 配置示例（如适用）
```

## 7. 回归测试策略

### 7.1 影响范围

以下变更需要触发规则链回归测试：

| 变更类型 | 影响范围 | 回归级别 |
|---------|---------|---------|
| 新增节点 | 该节点所在 Phase + 依赖该 Phase 的链 | 同级 |
| 修改 ChainEngine | ChainCoordinator + 所有链 | 全量 |
| 修改 ChainCoordinator | 所有多链策略 | 全量 |
| 修改 ChainWorkerPool | StrategyEvaluator 提交逻辑 | 跨组件 |
| 修改核心数据类型 | 序列化 + 所有链 | 全量 |

### 7.2 回归测试执行

```bash
# 全量回归
dotnet test backend/TradeX.Tests

# 指定受影响模块
dotnet test backend/TradeX.Tests --filter "FullyQualifiedName~Rules|FullyQualifiedName~Trading"
```

## 8. 已知限制与后续计划

### 8.1 当前覆盖缺口

- `ChainEngine` 执行流程（多阶段交互）尚无独立测试
- `ChainCoordinator` 决策合并逻辑尚无独立测试
- `ChainWorkerPool` 并发调度尚无测试
- 性能/压力测试尚未覆盖

### 8.2 后续计划

1. **补充 ChainEngine 单元测试** — 覆盖 7 个阶段的错误策略和正常流程
2. **补充 ChainCoordinator 单元测试** — 覆盖 10+ 种合并场景
3. **补充 ChainWorkerPool 单元测试** — 覆盖双 Lane 调度和并发控制
4. **集成测试** — 验证完整评估链路（信号 → 规则链 → 风控 → 执行）
5. **性能基准测试** — 评估 200 节点/100 链压力下的吞吐量
6. **模糊测试** — 随机生成 JSON 参数验证节点健壮性
