## 后端

### BE-001: Go 1.26+

| 属性 | 内容 |
|------|------|
| **选型理由** | Go 编译效率高、并发性能出色（Goroutine/Channel）、部署简单（单二进制）、静态类型减少运行时错误，适合构建高性能 API 服务 |
| **备选方案** | Node.js（不适合 CPU 密集型，单线程模型）、Java（启动慢，构建复杂）、Python（性能瓶颈，GIL 限制） |
| **版本说明** | Go 1.26 是截至 2026 年的最新稳定版，持续优化编译器性能、改进静态分析工具链、完善泛型体验。1.24 起内置 `slog` 标准库日志、`testing/fstest` 改进，1.22 起 for 循环变量语义修正避免闭包陷阱 |
| **已知约束** | Go 的泛型在 1.18 引入，但整体泛型能力仍不如 Java/C# |
| **依赖服务** | — |

### BE-002: Gin Web 框架

| 属性 | 内容 |
|------|------|
| **选型理由** | Go 生态中最流行的 HTTP 框架，性能极高（基于 httprouter），中间件机制成熟，社区活跃，文档丰富 |
| **备选方案** | Chi（轻量但生态弱）、Echo（功能类似但社区较小）、Fiber（类 Express API，但使用 fasthttp，兼容性风险） |
| **版本说明** | v1.10 起支持 `TrustedProxies`、路由分组改进，稳定无 Breaking Change |
| **已知约束** | 内置的 Validator 功能有限，需配合 `go-playground/validator` |
| **依赖服务** | — |

### BE-003: Ent ORM

| 属性 | 内容 |
|------|------|
| **选型理由** | 类型安全的 ORM，代码生成机制确保编译时发现错误；原生支持 GraphQL 集成、缓存、审计钩子；Schema as Code 设计理念，模型定义即文档 |
| **备选方案** | GORM（反射驱动，运行时错误不易发现，N+1 问题常见）、sqlx（手写 SQL，缺少迁移能力）、GORM Gen（代码生成但社区小） |
| **版本说明** | 推荐 latest 稳定版，v0.14+ 已支持 SQLite、Gremlin 等多方言 |
| **已知约束** | 复杂查询（如窗口函数、CTE）需要手写 SQL 扩展；社区相对 GORM 小 |
| **依赖服务** | PostgreSQL 16+ |
| **ADR 关联** | ADR-002 |

### BE-009: Casbin

| 属性 | 内容 |
|------|------|
| **选型理由** | Go 生态中最成熟的访问控制库，支持 ACL、RBAC、ABAC 等多种授权模型；基于配置文件定义策略，无需修改代码即可调整权限规则；性能优异，策略匹配可在微秒级完成 |
| **备选方案** | 手写中间件实现授权（硬编码，不可维护）、`go-admin` 内置授权（耦合度高）、OAuth 2.0 Scope（仅适用于外部 API 授权） |
| **版本说明** | `github.com/casbin/casbin/v2` latest 稳定版，v2 支持多适配器（MySQL/PostgreSQL/RBAC with deny/超级管理员） |
| **已知约束** | 策略变更通常需要调用 `LoadPolicy()` 重新加载（支持 Watcher 自动同步但需额外配置）；学习曲线中等，需理解模型配置文件语法 |
| **依赖服务** | PostgreSQL（策略存储适配器） |

### BE-010: Redis 7+

| 属性 | 内容 |
|------|------|
| **选型理由** | 业界最流行的内存数据存储，支持字符串、哈希、列表、集合、有序集合、流等多种数据结构；读写性能极高（~100k ops/s）；在项目中用于缓存高频查询结果、维护 JWT Token 黑名单（实现主动失效）、限流计数器、以及异步任务队列 |
| **备选方案** | Memcached（仅支持简单 KV，无持久化，无数据结构）、本地内存缓存（进程隔离，无法共享，重启丢失）、etcd（分布式一致性强但性能不如 Redis） |
| **版本说明** | 7+ 引入 ACL v2、Sharded Pub/Sub、函数特性增强，推荐 7.2+ |
| **已知约束** | 单线程事件循环，单个大 Key（big key）会阻塞其他操作；内存为主，持久化能力弱于数据库；无自动水平分片（需 Redis Cluster 或代理层） |
| **依赖服务** | — |

### BE-012: dongle

| 属性 | 内容 |
|------|------|
| **选型理由** | `github.com/golang-module/dongle` 是 Go 生态中功能全面的编解码与加解密工具库，与 Carbon 同属 golang-module 生态；提供 Base64、Hex、MD5、SHA1/256/512、HMAC、AES、DES、RSA、bcrypt 等 30+ 种编码/加密操作的统一 API，一套 API 即可覆盖密码哈希、数据加密、JWT 签名验证等常见场景，显著减少样板代码 |
| **备选方案** | Go 标准库 `encoding/base64` `crypto/sha256` `crypto/aes` `golang.org/x/crypto/bcrypt` 等（API 零散，多步组合需大量样板代码） |
| **版本说明** | `github.com/golang-module/dongle` latest 稳定版，0.1+ 起已覆盖主流编解码和加解密方案 |
| **已知约束** | 仅提供编码/加密操作方法封装，不管理密钥生命周期；JWT 核心签发与验证逻辑仍由应用层编排，dongle 负责签名/验签的哈希运算环节 |
| **依赖服务** | — |

### BE-013: OpenTelemetry

| 属性 | 内容 |
|------|------|
| **选型理由** | 行业标准可观测性框架，统一链路追踪（Tracing）、指标（Metrics）、日志（Logs）三大信号；Go SDK 由官方维护，与 Gin/ent/zerolog 等生态集成成熟；支持 OTLP 协议导出到 Jaeger、Prometheus、Grafana 等后端 |
| **备选方案** | Prometheus SDK 裸用（仅有指标，无追踪）；Jaeger Client（已弃用，推荐 OTel）；Sentry（SaaS 绑定，不通用） |
| **版本说明** | `go.opentelemetry.io/otel` latest 稳定版，推荐 v1.28+；Gin 集成使用 `go.opentelemetry.io/contrib/instrumentation/github.com/gin-gonic/gin/otelgin` |
| **已知约束** | 引入额外依赖约 10+ 个传递包；OTLP exporter 需配合 Collector 或后端地址配置；zerolog 与 OTel 日志关联需手动桥接 |
| **依赖服务** | Jaeger / Tempo（追踪后端，可选）、Prometheus + Grafana（指标，可选） |

## 废弃/未采纳方案记录

| 方案 | 考虑阶段 | 未采纳原因 | 替代方案 |
|------|---------|-----------|---------|
| GORM | 架构设计 | 运行时反射导致类型不安全，N+1 问题高频出现，迁移体验不如 Ent | Ent |
| MySQL | 架构设计 | JSON 支持和窗口函数弱于 PostgreSQL | PostgreSQL |
| Session + Redis | 架构设计 | 需要集中存储，不利于水平扩展 | JWT |
| Element Plus | 架构设计 | 设计风格偏旧，与 Vue 3 的组合式 API 配合不如 Arco 顺滑 | Arco Design Vue |
| npm | 架构设计 | 安装速度慢，lockfile 冲突频繁 | bun |
| Chart.js | 架构设计 | 图表类型少，不支持大数据量的流畅交互 | ECharts |
